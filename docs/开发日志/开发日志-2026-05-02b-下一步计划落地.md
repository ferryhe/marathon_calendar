## P0 · 历史赛事自动归档

**变更文件**：
- `server/syncScheduler.ts` —— 新增 `archivePastEditions()` export，并在 `syncNowOnce()` 入口自动调用
- `server/routes.ts` —— 新增 `POST /api/marathons/archive-past` 公开端点

**SQL 逻辑**：

```sql
UPDATE marathon_editions
   SET registration_status = '已完赛'
 WHERE race_date < CURRENT_DATE - INTERVAL '1 day'
   AND registration_status NOT IN ('已完赛','已结束','已取消');
```

**手动归档结果**：dev/prod 各 12 条已过赛日的届次被改为「已完赛」（含巴黎/波士顿/伦敦/东京 等大满贯）。
后续每次 scheduler tick / 手动 sync 都会再跑一次，幂等。

**端点验证**：
```
POST /api/marathons/archive-past 200
{"data":{"archivedCount":0,"message":"无需归档：所有届次状态已是最新"}}
```

## P1 · Zuicool 大批量补绑

**SQL 文件**：`docs/数据库变更/2026-05-02e-zuicool-batch-binding.sql`

按计划文档的「4 步发现流程」对每个候选赛事：
1. 通过 zuicool /events 列表 + webSearch 拿到 event/{id}
2. webFetch 该页面，校验 H1 是否含目标年份 + 赛事名
3. H1 不符则**不绑**（如衡水湖 event/3587780 显示的是 2025、巴黎 event/64608 显示的是 2025）
4. 入 marathon_sources（second source，不覆盖 official）

**执行结果**：
- 候选 22 个、写入 21 个、严格校验否决 3 个（衡水湖、巴黎 因年份不符；某海外因 H1 显示历史届次）
- Zuicool bindings 由 **6 → 21**（净 +3 来自 P1 部分：beijing / dalian / tokyo-2027；其余 12 由 P2 引入的新赛事自带）
- changbaishan / linxia 没有 event/{id}，但有 reg.zuicool.com 报名直链，仍写入 marathon_sources

## P3 · runchina + shuzixindong（**受限放弃**）

按计划做了探活：

**runchina（田协）**：
- webSearch 多次返回的均是田协**赛事目录 PDF**的内容总结（"2026年全国预计举办 492 场"），无具体可抓 URL
- 田协官网的赛历入口是静态 PDF/Word 文件，无 API、无结构化 HTML 列表页
- 接入难度：需先写 PDF 解析器拿 492 个赛事名 + 标牌等级 + 日期，再 fuzzy match 到我们 49 个赛事
- 工作量评估：> 6h，远超计划估算的 3h

**shuzixindong（数字心动）**：
- 用 `for sub in {ningbo,beijing,shanghai,xiamen,guangzhou,shenzhen,hangzhou,wuhan,chengdu,chongqing,nanjing,suzhou}` 枚举 `${sub}marathon.shuzixindong.com`
- 结果：**只有 ningbo 返回 200，其余 11 个全部 DNS 失败**（HTTP 000）
- 进一步用 webSearch 查询「{城市}马拉松 数字心动 报名」，返回的页面均指向**数字心动 APP 内**（非 SSR），没有可绑的稳定 URL
- 结论：宁波是孤例，shuzixindong **不存在可批量补绑的子站集合**

**决定**：本会话不追加 runchina/shuzixindong 绑定。建议下次会话单独立项做 runchina PDF 解析。

## P4 · 数据健康度看板

**变更文件**：
- `server/routes.ts` —— `/api/admin/stats` 响应新增 `health` 字段（marathons/editions/bindings 三组指标）
- `client/src/lib/adminApi.ts` —— `AdminStats` 类型扩展
- `client/src/pages/AdminData.tsx` —— 概览 Tab 末尾新增「数据健康度」卡片（6 格 grid）

**指标**：

| 指标 | 含义 | 当前 dev 值 |
|---|---|---|
| 赛事总数 / 含已发布届次 | 全量 + 至少一个届次 published 的赛事数 | 49 / 49 |
| 届次总数 / 已完赛 | 含 P0 归档结果 | 53 / 12 |
| 待公布日期 | `race_date IS NULL` 的届次数 | 0 |
| 报名中无 URL | 状态=报名中 但 `registration_url` 为空 | 0 |
| 第三方绑定 / 自动更新 | 总绑定数 + 未禁用自动更新的数量 | 60 / 待统计 |
| 陈旧绑定 | 自动更新但 `last_checked_at` 早于 30 天前 | 持续观察 |

**Schema 适配踩坑**：
- `marathons` 表**没有** `publishStatus`（只有 editions 有）→ 用 `SELECT COUNT(DISTINCT marathon_id) FROM marathon_editions WHERE publish_status='published'` 派生
- `marathon_sources` **没有** `autoUpdateEnabled` 列 → 由 `nextCheckAt <> AUTO_UPDATE_DISABLED_NEXT_CHECK_AT (2099-12-31)` 哨兵值派生
- `marathon_sources` **没有** `lastSyncedAt` 列 → 用 `lastCheckedAt`

**测试 ID**：每个指标格子带 `data-testid="text-health-{指标}"`，便于回归。

## 增量 · 2026-05-02f · marathonbm.com 探源 + 10 新马拉松

**触发**：用户 "https://www.marathonbm.com/event 从这里找一些放进列表"

### 探源结论

| 项 | 结论 |
|---|---|
| URL 结构 | SPA（Vue），列表页路径 `/event`，详情页 `/eventDetail/{id}` |
| 后端网关 | `gateway.marathonbm.com`（仅在 JS bundle 暴露），未公开 list API；探活 `/api/event/list` `/api/race/list` `/api/event/page` 等 30+ 路径全 404 |
| 总规模 | 列表共 **398 页**，按区域/项目/年份/赛况/月份筛选 |
| 内容混杂 | 大量越野赛/泡泡跑/虎符挑战赛等非主流赛事，需按 "马拉松" 关键字过滤 |

### 路径

1. **直接抓** marathonbm.com/event 第 1 页 → 仅拿到 2 个未收录马拉松：**临泽**、**镇宁黄果树半程**（其余要么是已收录如乌兰察布/临夏，要么是越野/泡泡类被排除）。
2. **横向扩展** webSearch 找尚未收录的省会马拉松，逐个 webFetch 验证 zuicool H1：
   - ✅ 哈尔滨 / 长春 / 郑州 / 福州（zuicool/event H1 全部命中）
   - ✅ 沈阳 / 临泽 / 银川 / 长沙 / 南宁（H1 解析或第二轮搜索补全 event ID）
   - ❌ 济南（zuicool/event/4713887 → 实际是 "古蜀风华成都线上马拉松"，废弃）
   - ❌ 海口（reg.zuicool.com/17990 仅是跨年报名页，无独立 2026 届）
   - ❌ 贵阳（race_date 4/8 已过）
3. **生成 SQL**：[2026-05-02f-marathonbm-batch.sql](../数据库变更/2026-05-02f-marathonbm-batch.sql)
